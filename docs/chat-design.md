## 1. 概要

SSG ONEのLINE Botにコンテキストを持つAIチャット機能を追加する。
顧客向けと従業員向けで異なる人格・権限・機能を持つ2種類のBotを実装する。

**技術スタック**
- フロー制御: LangGraph（Python）
- LLM: Claude API（Anthropic）
  - 成形判断・アラート判定: claude-haiku（軽量・低コスト）
  - 本命応答: claude-sonnet
- 実行環境: AWS Lambda（Python）

---

## 2. アーキテクチャ

```
顧客向けLINEアカウント          従業員向けLINEアカウント
        ↓                                ↓
  Go Lambda                        Go Lambda
（Webhook受信・振り分け）       （Webhook受信・振り分け）
        ↓                                ↓
        └──────────────┬─────────────────┘
                       ↓ AWS SDK Lambda invoke
              Python Lambda（LangGraph）
              ├── 顧客Botモジュール
              └── 従業員Botモジュール
                       ↓
          Supabase（履歴読み書き）／Claude API
```

**方針**
- 店舗ごとに独立したLINEアカウント＋Go Lambdaをデプロイ
- Python Lambdaは現在1つ。顧客・従業員を内部モジュールで分離し、将来の分割に備える
- Go → Python Lambda間の呼び出しはAWS SDK経由（Lambda直接invoke）
- 環境変数 `STORE_ID` で店舗を特定

## 3. ユーザー振り分け

LINEメッセージ着信時に `staff.line_user_id` と照合して自動振り分け。

```
LINEメッセージ着信
  ↓
staff.line_user_id に一致する？
  ├─ YES → 従業員Bot
  └─ NO  → 顧客Bot
```

## 4. キャラクター・人格設定

| | 顧客Bot | 従業員Bot |
|--|---------|----------|
| 口調 | 丁寧・温かみ | 丁寧・寄り添い |
| 名前 | 〇〇店のアシスタント | 〇〇店のアシスタント |
| ターゲット | 女性顧客メイン | スタッフ全般 |

**名前の取得**
`STORE_ID` 環境変数 → `stores.store_name` をDBから取得してプロンプトに埋め込む。

## 5. 会話履歴

| 項目 | 設定値 |
|------|--------|
| 保持件数 | 直近20件 |
| セッションリセット | 最終メッセージから7日経過で新規セッション |
| セッションID | UUID（UNIQUE制約あり） |
| 手動リセット | なし（将来追加可能） |

**セッションIDの管理**
- 最終メッセージの `created_at` を確認し、7日以上経過していれば新規UUIDを発行
- UUID衝突対策としてDBに UNIQUE制約、アプリ側で挿入失敗時はリトライ

| created_at | TIMESTAMPTZ | NO | NOW() |

## 7. LangGraphグラフ設計

```
[セッション確認]
      ↓
[履歴取得]（直近20件）
      ↓
[成形判断]（Haiku）
  誤字・脱字・意図の不明瞭さを検知
      │
      ├─ 補正必要 → [ユーザーに確認返信] → 終了
      │
      └─ 補正不要
              ↓
      [意図判断・Function選択]（Sonnet）
              │
              ├─ DBデータが必要 → [DB取得] ─┐
              │                              │
              └─ 不要 ─────────────────────┘
                                             ↓
                                    [Claude本命呼び出し]（Sonnet）
                                             ↓
                                    [アラート判定]（Haiku）※従業員Botのみ
                                             ↓
                                    [履歴保存]
                                             ↓
                                    [LINE返信]
```

## 8. 顧客Bot設計

### 回答範囲

**対応するもの**
- 店舗情報（営業時間・場所・電話番号）→ システムプロンプトに静的埋め込み
- メニュー・料金の案内
- 施術内容の説明
- 予約の相談（予約システム実装後は連携）
- 一般的な健康・ウェルネスに関する質問
- 体の不調・疲労の相談 → 共感して来店を案内

**フィルタするもの**
- 店舗と無関係な話題 → 話題を戻す
- ネガティブな話題（クレーム・悪口など）→ スタッフへの誘導
- 医療的な診断・アドバイス
- 競合他社との比較

### クレーム対応
Botでは対応せず「スタッフに直接お声がけください」と誘導。

## 9. 従業員Bot設計

### 回答範囲

**個人系**（全ロール共通）
- 自分の売上・件数
- 自分の勤怠（出退勤・残業時間）
- 自分の担当顧客情報

**店舗系**（store_manager以上）
- 店舗全体の売上状況
- スタッフ別実績

**全店系**（regional_manager以上）
- 全店舗の売上・実績

### DB参照方式
Function Callingで質問内容から必要なデータのみ動的取得。

| Function | 説明 | 必要権限 |
|----------|------|----------|
| `get_my_sales` | 個人売上取得 | 全ロール |
| `get_my_attendance` | 個人勤怠取得 | 全ロール |
| `get_store_sales` | 店舗売上取得 | store_manager以上 |
| `get_store_staff_performance` | スタッフ別実績 | store_manager以上 |

### カウンセリング・アラート機能

**ネガティブ対応**
体調不良・職場の悩みなどはカウンセリング的に受け止めて寄り添う。

**アラート判定**
プロンプトテンプレートに「検知すべきニュアンス」を定義。ネガティブ検知はその一例で、将来他のパターンも追加可能。

```
以下のニュアンスを検知した場合はアラートフラグを立ててください：
- 強いストレス・疲弊の表現
- 職場への不満・人間関係の深刻な悩み
- （将来追加可能）
```

**アラート通知**
- 通知先: 環境変数 `OWNER_LINE_USER_ID` で設定（将来複数人対応可能な構造）
- 通知手段: 既存の LINEプッシュ通知（`PushHandler`）を流用
- 発言した従業員への通知: なし
- 前提: 利用規約・事前説明でAIによる判定があることをユーザーに周知

## 10. 将来の分割方針

現在はPython Lambdaを1つとし、顧客・従業員を内部モジュールで明確に分離して実装する。
分割が必要になった際は以下の構成に移行する。

```
顧客向けLINEアカウント → 顧客Go Lambda → 顧客Python Lambda
従業員向けLINEアカウント → 従業員Go Lambda → 従業員Python Lambda
```

外部公開が必要になった場合は、Lambda invokeのラッパーとなる別Lambdaを立てて対応する。
