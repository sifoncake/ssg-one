# SSG ONE - マルチストア・ウェルネス事業管理プラットフォーム

## プロジェクト概要

**SSG ONE**は、日本国内の複数店舗展開ウェルネス事業向けに設計されたフルスタックSaaSプラットフォームです。顧客向けのLINE配信と社内向けのAI支援を組み合わせ、複数店舗にまたがるシームレスな業務運営管理を実現します。

**エレベーターピッチ:** リアルタイム分析、AI支援の業務オペレーション、高度なロールベースアクセス制御を組み合わせたクラウドネイティブな業務管理ソリューション。複数店舗を展開するウェルネス事業の運営を効率化します。

**ターゲットユーザー:** 複数店舗を運営するウェルネス事業者（マッサージ、鍼灸、ボディケア）で、スタッフ・顧客・売上・店舗間オペレーションの一元管理を必要としている企業。

**解決した課題:** 店舗間のデータサイロと手動連携を解消。リアルタイム可視性、自動顧客エンゲージメント、セキュアなマルチテナントデータ管理を備えた統合プラットフォームを提供。

## 実装した主要機能

### 管理ダッシュボード
- 月次比較機能付きリアルタイム店舗パフォーマンス指標
- 月間目標に対する売上進捗の視覚的表示
- ドリルダウン可能なクロスストア分析
- 過去データ対応の動的期間選択

### ロールベースアクセス制御
柔軟な4階層権限システムを実装:

| ロール | アクセス範囲 | 権限 |
|--------|--------------|------|
| システム管理者 | 全店舗 + 設定 | プラットフォーム全権限 |
| 全店管理者 | 全店舗 | 店舗横断オペレーション |
| 店舗管理者 | 配属店舗 | スタッフ・取引管理 |
| 一般従業員 | 配属店舗 | 基本操作 |

設計上のポイント: ロールは店舗配属レベルで割り当てられ、同一ユーザーが異なる店舗で異なる権限を持つことが可能。

### マルチストアデータ管理
- スタッフの異動・応援配属の追跡
- 顧客の店舗横断来店履歴
- 複数店舗での回数券利用
- 店舗別月間目標とパフォーマンス指標

### LINE運用（顧客向け）
- Webhookベースのリアルタイムメッセージ処理
- キーワード検出による管理者アクセストリガー
- 一斉送信・特定ユーザーへのプッシュ配信

### AI支援（社内向け）
- LINE運用の社内アシスト（Claude API連携）

### 認証システム
- マジックリンクの生成と検証
- デバイスフィンガープリントによる端末認識
- 新規デバイス向け6桁2FAコード
- アプリ内認証のためのLIFF IDトークン検証
- 有効期限10分の使い捨てトークン

### 売上・顧客管理
- 商品タイプ分類付き取引記録
- 複数支払方法対応
- スタッフ別売上貢献度
- 顧客生涯価値（LTV）追跡
- 顧客セグメンテーション（VIP、複数店舗、常連、新規）

## アーキテクチャと設計判断

### サーバーレスアーキテクチャ
**判断:** バックエンドサービスにAWS Lambda + API Gatewayを採用

**理由:**
- インフラ管理ゼロ
- Webhookトラフィックスパイクへの自動スケーリング
- 変動トラフィックに最適な従量課金モデル
- Goランタイムによるサブ秒コールドスタート

### マルチテナントデータベース設計
**判断:** 単一データベースを前提としたマルチテナント設計

**理由:**
- テナント別データベースより保守が簡単
- マルチテナント前提のデータ分離設計
- 効率的な店舗横断分析クエリ
- 初期スケールに適したコスト効率

### マジックリンク認証
**判断:** 2FAフォールバック付きLINE経由パスワードレス認証

**理由:**
- 既存のLINE ID活用（日本では普及率が高い）
- パスワード管理の負担解消
- デバイスフィンガープリントで信頼デバイスの2FA省略
- セキュアなトークンライフサイクル（10分有効・使い捨て）

### 型安全な開発
**判断:** TypeScriptフロントエンド + Go バックエンドで厳密な型付け

**理由:**
- コンパイル時のエラー検出
- 自己文書化されるAPIコントラクト
- 本番環境でのランタイムエラー削減
- IDE補完による開発体験の向上

### ハイブリッドAPIアーキテクチャ
**判断:** Next.js APIルート + Lambda関数の併用

**理由:**
- Next.jsルートによるローカル開発の簡便性
- Lambdaによる本番スケーラビリティ
- 環境ベースルーティングでシームレスな切り替え
- VercelでのフロントエンドからAPI呼び出しのレイテンシ削減

## 克服した技術的課題

### 複雑なマルチストアデータモデリング
**課題:** スタッフは複数店舗で異なるロールで勤務可能、顧客は複数店舗を利用。

**解決策:** 時間的追跡（開始/終了日）とロール別配属設計を備えた中間テーブル（`staff_store_assignments`、`customer_store_history`）を実装。データベース制約でロールと店舗の組み合わせの整合性を保証。

### LINE経由のリアルタイム認証
**課題:** 従来のログインフォームなしでLINEチャットから安全な管理者アクセスを実現。

**解決策:** 多段階認証フローを設計:
1. キーワード検出でトークン生成
2. マジックリンク + 2FAコードをLINE返信で送信
3. デバイスフィンガープリント付きブラウザ検証
4. アプリ内シナリオ向けLIFF IDトークン検証
5. デバイス信頼に基づく条件付き2FA

### 型安全なSupabase連携
**課題:** Supabaseは動的型を返却、複雑なJOINでの型安全性確保。

**解決策:** すべてのデータベースレスポンスに明示的なTypeScriptインターフェースを作成。Supabaseクエリ結果に型ガードと明示的キャストを実装。厳密な型定義ですべての`any`型を排除。

### 年度をまたぐ日付範囲クエリ
**課題:** 月選択で年の境界と動的な日付範囲を処理する必要。

**解決策:** 月ごとの日数の違いを考慮した動的月末計算を実装。適切なタイムゾーン処理でパラメータ化された日付クエリを使用。初期ロード時にデータが存在する最新月を自動検出。

### 過去データとライブ更新の両立
**課題:** ダッシュボードは過去月を表示しつつ、最新データ状態を反映する必要。

**解決策:** データ取得とディスプレイロジックを分離。参照整合性を維持しながら選択に基づいてデータをクエリする月セレクターを実装。利用可能なデータがある最新月を検出する初期化ロジックを追加。
